# Security Audit: Forward Secrecy & Traffic Decryption Risk
**Сервер:** ikenibornvpn (80.209.240.162 / ft.ikeniborn.ru)  
**Дата:** 2026-04-16  
**Метод:** Black-box + white-box: tcpdump + tshark с приватными ключами сервера (LE + Reality)

---

## Executive Summary

**Угроза расшифровки захваченного трафика — НЕРЕАЛИЗУЕМА.**

Злоумышленник, получивший любой из приватных ключей сервера (TLS-сертификат Let's Encrypt или Reality X25519 ключ), **не сможет расшифровать** перехваченный ранее или будущий трафик пользователей. Это гарантируется архитектурой TLS 1.3 и протоколом VLESS Reality.

Проверено практически: tcpdump-захваты на портах 443, 1080, 8118 и внутреннем loopback-порту 8443 (между nginx и xray) не поддаются расшифровке ни одним из доступных ключей.

---

## 1. Конфигурация TLS (nginx)

| Параметр | Значение | Оценка |
|---|---|---|
| Протокол | **TLSv1.3 only** | ✅ Безопасно |
| Cipher suite | **TLS_AES_256_GCM_SHA384** | ✅ AEAD + PFS |
| Тип сертификата | **ECDSA P-256** (не RSA) | ✅ Нет RSA-уязвимости |
| Key exchange | **x25519 (ECDHE)** — подтверждено tshark | ✅ Forward Secrecy |
| Порты TLS | 1080, 8118, 8448, 4443 | ✅ TLS 1.3 на всех |

### Почему TLS 1.3 делает расшифровку невозможной

TLS 1.3 **обязывает** использовать ECDHE для каждого сеанса:
1. При каждом рукопожатии клиент и сервер генерируют **одноразовые (эфемерные) ключи** — независимо от статических
2. Из них выводится **сессионный ключ** — нигде не хранится, уничтожается после сеанса
3. Статический приватный ключ сервера используется **только для аутентификации** (подписи), не для шифрования
4. Без эфемерного ключа конкретной сессии расшифровка невозможна математически

## 2. Тест 1 — LE сертификат (порты 443/1080/8118)

### Методология
- Захвачено **1 246 пакетов** на портах 1080/8118 за 30 сек (954 КБ)
- Захвачено **409 пакетов** на порту 443 за 20 сек (77 КБ)
- Попытка расшифровки через `tshark` с реальным `privkey.pem` (LE-сертификат)

### Результат

```
tshark: Can't load private key from privkey.pem:
  private key public key algorithm isn't RSA
```

Расшифровка не состоялась по двум причинам одновременно:
1. **ECDSA-ключ** — LE-сертификат использует ECDSA P-256, а не RSA; инструменты RSA-расшифровки его не поддерживают
2. **TLS 1.3 / x25519** — tshark подтвердил `Key Share Entry: Group: x25519` в Server Hello; даже при RSA-ключе TLS 1.3 не позволяет расшифровать трафик по статическому ключу сервера

`tls.app_data` в дампе — **зашифрованный payload** (hex), не открытый текст.

## 3. Тест 2 — Reality X25519 ключ (loopback порт 8443)

### Архитектура трафика

```
Client:443 → nginx (ssl_preread, БЕЗ расшифровки) → 127.0.0.1:8443 (xray Reality)
```

nginx не терминирует TLS — только читает SNI и проксирует сырые TLS-пакеты. На порту 8443 идёт полноценный TLS/Reality handshake непосредственно с xray.

### Как Reality использует статический X25519 ключ

```
Клиент знает: server_publicKey (из конфига клиента)
Сервер знает: server_privateKey (32 байта X25519, из xray_config.json)

Handshake:
  1. Клиент генерирует ЭФЕМЕРНЫЙ x25519 ключ (одноразовый)
  2. DH(client_ephemeral_priv, server_pub) → аутентификация сервера
  3. TLS 1.3 key_share: оба генерируют СВОИ эфемерные ключи для сессии
  4. session_key = ECDHE(client_ephemeral, server_ephemeral)
                   ↑ не server_privateKey из конфига!
```

**Статический `privateKey` используется только для аутентификации клиентом, не для шифрования данных.**

### Результат

- Захвачен трафик на `lo:8443` (15 пакетов)
- Reality X25519 ключ — 32 байта, алгоритм X25519 (не RSA, не ECDSA)
- tshark/ssldump не поддерживают X25519 static key для TLS-расшифровки
- Даже при поддержке: TLS 1.3 гарантирует PFS через эфемерный key_share

**Расшифровать дамп с loopback 8443 с помощью Reality privateKey невозможно.**

## Заключение

Сервер **корректно защищён** от атак расшифровки трафика через компрометацию приватных ключей. Ни LE-сертификат (ECDSA P-256), ни Reality X25519 ключ не позволяют расшифровать захваченный трафик — ни снаружи (порт 443), ни изнутри контейнера (loopback 8443). TLS 1.3 с ephemeral x25519 key_share обеспечивает **полный Forward Secrecy** на уровне протокола.

Единственный реальный вектор — прямой root-доступ к серверу в момент активной сессии (дамп RAM процесса или SSLKEYLOGFILE). Защита от этого выходит за рамки конфигурации VPN и требует hardening на уровне ОС.
